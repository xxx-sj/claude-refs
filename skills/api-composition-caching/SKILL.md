# API 조합 · 캐시 · 개인화 경계 프레임워크

새 조회 API를 만들 때 **"한 API에 합칠지 / 내부에서 병렬로 나눌지 / 캐시는 어디에 둘지 / 개인화 값을 어떻게 붙일지"** 를 명시적으로 결정한다.

> **기준은 "병렬로 가져올 수 있으니 쪼갠다"가 아니다. 경계(boundary)가 같은 것끼리 묶고, 다른 것만 쪼갠다.**

핵심 불변식 한 줄: **개인화 값(`is_liked`, 좋아요, `followed_by_me` 등)은 절대 캐시된 payload 안에 넣지 않는다. 캐시는 익명·공유 가능한 것만 담고, 개인화는 캐시 밖에서 매 요청 fresh로 붙인다.**

이 스킬은 의사결정 가이드다. 답을 강요하지 말고, 아래 단계로 사용자의 케이스를 같이 분류해준다.

---

## 1. 데이터를 가르는 4개의 경계

응답에 들어갈 조각 A와 B를 같은 응답에 담을지는 아래 4축으로 판단한다. **하나라도 다르면 쪼개는 쪽**으로 기운다.

| 축 | 같이 담아도 되는 신호 | 쪼개야 하는 신호 |
|---|---|---|
| **캐시 수명** | 둘 다 정적(분~시간) | 한쪽이 초 단위로 변함 |
| **개인화** | 둘 다 익명(모두 동일) | 한쪽이 유저별(`is_liked`, 좋아요) |
| **소유 도메인** | 같은 모듈/서비스 | 다른 모듈/팀 소유 |
| **실패 허용** | 둘 다 must(없으면 화면 X) | 한쪽은 nice-to-have(없어도 화면 뜸) |

가장 자주 걸리는 축은 **개인화 × 캐시 수명**이다. 익명·정적인 본문과 유저별·휘발성인 좋아요를 한 응답에 섞으면, 응답 전체가 "가장 캐싱 안 되는 조각" 기준으로 떨어진다(= 캐싱 불가).

예: 상품 상세 화면 = 상품정보(익명·정적·must) + 좋아요 여부(유저별·휘발성·nice) + 판매자정보(익명·정적·다른 도메인). 좋아요만 경계가 다르므로 그것만 떼어낸다.

## 2. 두 레벨의 "나누기"

"나눈다"는 결정이 **두 군데**서 일어난다. 헷갈리지 말 것.

```
레벨 A (경계)   → HTTP 엔드포인트를 가른다     ← §1의 4경계로 판단
레벨 B (의존성) → 한 엔드포인트 안 조회를 가른다 ← 데이터 의존성으로 판단
```

### 레벨 B: 한 엔드포인트 안에서 병렬 vs 순차

레벨 A를 통과해 "같이 담기로" 한 조각들 사이에선 기준이 **데이터 의존성** 하나다.

- 앞 조회의 출력이 필요 없다 → **병렬** (동시 fan-out)
- 앞 조회의 출력이 입력으로 들어간다 → **순차**

```
// 상품정보(익명)와 좋아요(개인화)는 서로 의존 없음 → 병렬
const [product, liked] = await Promise.all([
    getProduct(productId),               // 익명·정적
    getLikeStatus(productId, userId),    // 개인화·휘발성
])

// 판매자정보는 product.sellerId 가 필요 → 순차
const seller = await getSeller(product.sellerId)

return { ...product, seller, liked_by_me: liked }
```

> ⚠️ **스택별 함정**: "나열만 하면 병렬"이 아닌 환경이 많다. 동기 코드는 그냥 순차 실행이고, 일부 워크플로 엔진은 명시적으로 감싸야 병렬이 된다. 해당 스택의 **명시적 병렬 구문**(JS `Promise.all`, Go `errgroup`/goroutine, 워크플로 엔진의 parallel primitive 등)을 의식적으로 쓴다.

## 3. 개인화는 캐시 경계 밖 (가장 중요한 룰)

캐시된 응답 안에 개인화 값을 박으면 둘 중 하나가 터진다:

- **전역 캐시**면 → A유저의 좋아요 상태가 B유저에게 노출 (정합성 사고)
- 그걸 막으려 **유저별 캐시 키**로 나누면 → 캐시 엔트리가 유저 수만큼 폭발 (히트율 붕괴)

그래서 개인화 값은 **항상 캐시 밖에서 매 요청 주입**한다:

```
// ❌ 나쁨 — 개인화를 캐시에 넣음
cache.set(`list:${listId}`, items.map(i => ({ ...i, liked_by_me: ... })))

// ✅ 좋음 — 캐시엔 익명만, 개인화는 꺼낸 뒤 주입
let items = await cache.get(`list:${listId}`)          // 익명·공유 가능
if (!items) { items = await buildList(); cache.set(`list:${listId}`, items) }

const likedSet = await getLikedIds(userId, items.map(i => i.id))  // 캐시 밖, 매 요청
return items.map(i => ({ ...i, liked_by_me: likedSet.has(i.id) }))
```

### 전역 캐시 vs 유저별 캐시 — 무엇으로 가르나

기준은 **"리스트의 내용(어떤 항목이 나오나) 자체가 유저마다 다른가?"**:

| 상황 | 캐시 전략 | 예 |
|---|---|---|
| 내용이 모두에게 동일 | **전역 캐시 키** + 개인화 캐시 밖 주입. 엔트리 1개, 히트율 최고. **1순위** | 주간 추천, 베스트, 카테고리 목록 |
| 내용 자체가 유저별 | **유저별 캐시 키**. 단 payload는 최소(id 목록만)로 | 팔로우 피드, 맞춤 추천 |

```
전역 캐시 키 :  home:weekly-picks:{week}           // 누구에게나 동일 → 1개 엔트리
유저별 캐시 키:  feed:following:{user_id}            // 내용이 유저별 → payload는 id 목록만
```

> ❌ **금지**: 좋아요 같은 **부가** 개인화 필드 때문에 캐시 키를 유저별로 나누는 것. 그건 캐시 밖 주입으로 푼다. 유저별 키는 **리스트 콘텐츠 자체가 개인화일 때만** 쓴다.

## 4. API를 쪼갤지는 "캐시가 어디 있느냐"가 결정한다

§3의 "캐시 밖 주입"을 구현하는 방법이 두 가지고, 어느 쪽이냐는 **캐시 위치**로 갈린다.

| 캐시 위치 | 권장 | 이유 |
|---|---|---|
| **origin 내부 (Redis / 인메모리 등)** | **한 API + 캐시 후 주입** | 어차피 모든 요청이 origin까지 옴. 쪼개면 클라 왕복만 2배, 이득 0 |
| **엣지 / CDN (origin 우회)** | **API 분리** (익명 라우트 + 개인화 라우트, 클라 병렬 머지) | CDN은 "모두에게 동일한 응답"만 캐싱 가능. 개인화 섞이면 캐싱 불가 |

**판단 규칙**: 캐시가 origin 내부(Redis)면 **한 API가 정답**. API 분리는 "익명 응답을 CDN 엣지에 올려 origin을 털겠다"는 결정이 생길 때만 한다.

```
// origin 내부 캐시 → 한 API
GET /products/:id      → [origin] Redis(상품) + 좋아요 fresh 병합 → 응답   (클라 왕복 1회)

// CDN 엣지로 옮길 때만 분리
GET /products/:id            → [CDN]    익명 상품, 모두 동일 → 엣지 캐시 ✅
GET /products/:id/like-state → [origin] 유저별, 항상 origin
// 클라가 둘을 병렬 호출해 product_id 로 머지
```

> 클라가 두 호출을 **병렬**로 쏘면 지연은 `max(A,B)`라 분리의 왕복 페널티는 거의 사라진다. 하지만 **"싸게 쪼갤 수 있다 ≠ 쪼갤 이유가 있다"** — 분리의 값어치는 캐시가 그 분리를 실제로 활용할 때(=CDN)만 생긴다.

### RTT 관점 (왜 서버 내부 병렬이 유리한가)

`전체 응답시간 = RTT(네트워크 왕복) + 서버 처리시간`. RTT(Round-Trip Time)는 요청→응답 한 바퀴 왕복 시간이고 거리가 결정한다.

| 경로 | RTT |
|---|---|
| 서버 ↔ DB/Redis (같은 DC) | ~0.1–1 ms |
| 모바일(LTE/5G) ↔ 서버 | ~30–100 ms |

→ 비싼 모바일 RTT는 **1번만**(클라 1회 호출) 내고, 싼 내부 RTT로 서버가 병렬 fan-out 하는 게 유리. "폰이 N번 따로 호출"보다 "폰 1회 + 서버 내부 병렬"이 빠르다. 그래서 캐시가 origin 내부일 땐 한 API + 내부 병렬이 기본.

## 5. 스케일 사다리 (조회 API)

앵커는 추측 트래픽이 아니라 **"익명 응답이 origin(서버+캐시)에 주는 부하"**. 가장 먼저 깨질 축을 본다.

- **T0**: 한 API + 내부 병렬. 캐시 없음.
- **T1** *(본문 enrichment가 무거워 P95 느려짐)*: **여전히 한 API.** 익명 부분만 Redis 캐싱 + 개인화는 캐시 밖 주입.
  - 토폴로지: origin 옆 Redis. 코드: 익명 조회 결과를 캐시 키에 저장 + 개인화 조회는 캐시 밖 병합 주입.
- **T2** *(origin 자체를 털고 싶다 / 글로벌 저지연)*: **API 분리 + CDN.**
  - 토폴로지: 익명 라우트 앞에 CDN, 개인화 라우트는 항상 origin.
  - 코드: 라우트 둘로 분리 → 익명 라우트에 `Cache-Control: public, max-age=…` → 개인화 조회를 상태 엔드포인트로 이동 → 클라가 키(예: `product_id`)로 병렬 머지.

임계치 숫자는 placeholder — 각 프로젝트 실측 P95/QPS로 교정.

> 주의: 응답 래핑 미들웨어가 모든 응답에 강제 `no-store`를 거는 코드베이스가 있다. 그런 경우 T1/T2에서 **해당 라우트의 캐시 헤더 우회가 선행**돼야 캐싱이 실제로 동작한다.

## 사전 체크리스트 (새 조회 API 작성 전)

```
□ 응답에 들어갈 조각들을 §1의 4경계로 분류했는가? (캐시수명/개인화/소유/실패허용)
□ 한 엔드포인트에 담기로 한 조각 중, 의존 없는 것을 명시적 병렬로 묶었는가?
□ 개인화 값(is_liked/좋아요/followed)을 캐시 payload에 넣지 않았는가?  ← 최우선
□ 캐싱한다면: 리스트 내용이 모두 동일(전역키) vs 유저별(유저키)을 구분했는가?
□ 유저별 캐시 키를 쓴다면, 그 이유가 "콘텐츠 자체가 개인화"여서인가? (부가 필드 때문이면 ❌)
□ API 분리를 고려 중이라면, 캐시가 CDN/엣지에 있는가? (Redis면 한 API 유지)
```

## 룰이 잡지 못하는 케이스 (인지 필요)

- 캐시 payload에 개인화가 섞였는지 — grep 자동 검출 어려움, 리뷰로 커버.
- "유저별 캐시 키 폭발" — 정상 동작처럼 보여 사고로 안 드러남. 캐시 키 설계 시 의식적으로 점검.
- 강제 `no-store` 같은 응답 미들웨어 때문에 캐싱을 켜도 효과 없는 코드베이스 — T1/T2 전에 우회 선행 필요.
