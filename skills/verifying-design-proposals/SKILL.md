# 설계 제안 검증 (Verifying Design Proposals)

## 개요

사용자가 설계 아이디어를 던지며 "확인해줘"라고 할 때 원하는 건 **동의가 아니라 실제 코드에 근거한 진실**이다. 현실에 비춰 검증하고, 틀렸으면 반대하라.

**핵심 원칙: 판단 전에 실제 코드를 추적하라. 검증(verdict)을 먼저, 동조(validation)는 나중에.**

## 규율

1. **실제 코드부터 추적.** 의견 내기 전에 grep/read 로 실제 구현·스키마·호출부를 본다. 기억으로 답하지 말고, 직접 찾을 수 있는 코드를 사용자에게 "어디 있냐"고 묻지 마라. 코드 안 보고 내는 판단은 추측이다.
2. **사실과 가정 분리.** 코드가 증명하는 것(`file:line`)과 추론을 구분해 말한다. 확인 못 한 건 명시.
3. **그들 안(案)만 채점하지 말고 비교.** 그들 안 포함 ≥2개 대안 — 후보 → 트레이드오프 표 → 택1 → 이유 + 못 막는 것 (선택 4단계). **그리고 "다른 대안은 *언제* 쓰는가"(각 안이 옳아지는 결정 신호)까지 말한다** — 지금 택한 것만이 아니라, 어떤 조건이면 다른 안이 정답이 되는지. 사용자가 1회성 답이 아니라 *재사용 가능한 판단 기준*을 얻게.
4. **동의 거부.** 결함이 있으면 그것부터 말한다. "좋은 생각이에요"로 시작하지 마라. 코드가 뒷받침하는 것만 인정. 틀린 yes 가 정직한 no 보다 비싸다.
5. **스코프 점검.** 지금 작업 범위 안인가, 아니면 별도/기존 관심사인가? 제안은 종종 작업 밖 공유 코드를 건드린다 — 짚어줘라.
6. **권고 + 결정 질문으로 끝.** 명확한 추천 하나 + 사용자가 정할 열린 질문.

## 위험 신호 — 실패 직전

- 코드 한 줄 안 읽고 답함 (이번 턴 grep/read 0회).
- "좋은 방향이에요, 근데…"로 시작.
- 그들 안만 평가, 대안 추적 없음.
- 스코프(범위 안/밖) 언급 없음.
- 직접 찾을 수 있는데 코드 위치를 사용자에게 물음.

## 합리화 표

| 변명 | 현실 |
|---|---|
| "패턴으로 답 가능" | 그 코드베이스의 특수성이 답을 바꾼다. 읽어라. |
| "제안이 합리적으로 들림" | 합리적 ≠ 그들 제약에서 옳음. 검증 후 판단. |
| "동의하면 진도 빠름" | 틀린 yes 가 정직한 no 보다 비싸다. |
| "당연히 스코프 안" | 대부분 제안은 공유/기존 코드를 건드린다. 확인. |
| "물어보는 게 빠름" | grep 가능하면 grep. 룩업을 떠넘기지 마라. |

## 쓰지 말 것

사소하거나 쉽게 되돌릴 선택(변수명, 이미 쓰는 lib), 또는 같은 턴에 관련 코드를 이미 추적한 경우 — 한 줄로 답한다.
