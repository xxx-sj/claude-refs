# 테스트 에이전트 설계 문서

> 두 개의 Claude Code 서브에이전트 — `test-writer`와 `test-reviewer` — 의 목적, 동작, 책임 경계를 정의한다.
> 이 문서는 사람이 리뷰하는 용도다. 실제 에이전트 파일은 이 설계를 따라 별도로 작성된다.

---

## 1. 개요

### 목적
테스트 코드의 **작성**과 **리뷰**를 명확한 신념 기반으로 일관되게 수행하는 에이전트 쌍을 만든다. 사람의 신념은 흔들리지만 에이전트의 시스템 프롬프트는 흔들리지 않는다는 특성을 이용해, 테스트 품질의 하한선을 코드로 강제한다.

### 위치 및 범위
- 위치: `~/.claude/agents/` (전역)
- 범위: 모든 프로젝트에서 자연어로 호출 가능 (에이전트 이름을 몰라도 됨)
- 모델: **둘 다 Opus 4.6**
- 자동화: `~/.claude/CLAUDE.md` 전역 규칙 + `/safe-commit` 슬래시 명령

### 두 에이전트의 역할 분담
- `test-writer` — 테스트 코드를 **직접 작성**한다. 파일 수정 권한 있음.
- `test-reviewer` — 기존 테스트를 **감사하고 리포트만** 한다. **Read-only**. 파일 수정 권한 없음.

이 분리의 핵심은 **부수효과의 격리**다. reviewer는 언제든 안전하게 호출할 수 있고, writer는 명확한 의도가 있을 때만 호출한다.

## 2. 공통 기반: 7가지 테스트 신념

두 에이전트가 공유하는 절대 규칙이다. writer는 이것을 따라 짜고, reviewer는 이것을 체크리스트로 평가한다.

### 신념 1. 트리아지 먼저
코드를 보자마자 "이 코드가 깨지면 무엇이 가장 아픈가"에 답하고 영향도순으로 우선순위를 매긴다. 모든 코드를 똑같이 다루지 않는다. 가장 아픈 것부터.

### 신념 2. 행동을 검증하지 구현을 검증하지 않는다
무엇이 일어나는가만 본다. private 메서드 직접 테스트 금지, 호출 횟수 단언 금지(부수효과가 명시적 주제가 아닌 한). 내부 리팩터링에도 살아남아야 한다.

### 신념 3. 진짜 객체를 우선, mock은 경계에만
같은 프로세스 안의 협력자는 진짜 객체로 구성한다. Mock은 네트워크/시간/랜덤/외부 서비스/I/O 경계에서만 사용한다. **내부적으로 구성 가능한 것은 절대 mock하지 않는다.**

### 신념 4. 결정성은 신성하다
flaky 금지. 시간 주입(`jest.useFakeTimers`, `setSystemTime`), 랜덤 시드 고정, 외부 상태 통제. 100번 중 1번 실패하는 테스트는 결국 무시되는 테스트다.

### 신념 5. 실패는 진단이 되어야 한다
테스트가 깨졌을 때 에러 메시지만 보고 원인이 보여야 한다. `expect(x).toBe(true)` 같은 의미 없는 단언 금지.

### 신념 6. 기존 컨벤션을 먼저 읽는다
**이 규칙이 다른 규칙보다 우선이다.** 같은 모듈의 기존 테스트 2~3개를 먼저 읽고 그 스타일을 정확히 따른다. 새 패턴을 임의로 도입하지 않는다. 일관성이 개인의 선호를 이긴다.

### 신념 7. 한글 테스트 이름 (it/test), 영문 식별자
- `it` / `test` 본문: 한글 명세 — `it('주문 금액이 음수면 거부한다', ...)`
- `describe` 블록: 영문 (클래스/모듈명 그대로)
- 변수, 함수, 헬퍼: 영문 (코드 컨벤션)
- 예외: 프로젝트가 이미 영문 테스트 이름을 일관되게 쓰면 그것을 따름 (신념 6이 우선)

## 3. 트리아지란?

### 용어의 기원
응급실 용어다. 환자가 한꺼번에 들어오면 의사는 "누가 먼저 죽는가"를 먼저 판단한다. 심정지 환자는 즉시, 골절은 나중에, 감기는 한참 뒤. **영향도순으로 우선순위를 매기는 행위**가 트리아지다.

### 테스트에서의 의미
> "이 코드가 깨지면 무엇이 가장 아픈가?"에 답하고 영향도순으로 테스트 우선순위를 매긴다.

### 예시 — 주문 서비스 트리아지

```
GoodsOrderService:
  createOrder()
  cancelOrder()
  formatOrderId()     ← 단순 문자열 포맷팅
  calculateRefund()
  getOrderById()

트리아지 결과:

[CRITICAL] cancelOrder     — 환불 계산 잘못 시 돈 손실
[CRITICAL] calculateRefund — 음수/0/소수점 처리 실패 시 사고
[HIGH]     createOrder     — 실패 시 매출 손실
[MEDIUM]   getOrderById    — 권한 검사 누락 시 IDOR (타 사용자 주문 노출)
[LOW]      formatOrderId   — 깨져도 표시만 이상함
```

### 왜 첫 단계여야 하는가
트리아지 없이 테스트를 짜면:
- 알파벳순으로 메서드 따라가며 중요한 것을 놓침
- happy path만 짜고 만족
- 모든 곳에 똑같이 5개씩 작성 → 시간 낭비
- 권한/멱등성/동시성 같은 "보이지 않는 위험" 누락

트리아지는 테스트 작성 전에 머릿속을 정리하는 5분짜리 사고 단계다. 이 5분이 나중의 5시간을 절약한다.

## 4. `test-writer` 에이전트

### 목적
주어진 파일/기능에 대해 신념 7개를 따르는 테스트 코드를 **직접 작성**한다.

### 도구 권한
`Read`, `Edit`, `Write`, `Glob`, `Grep`, `Bash` (Bash는 `yarn test` 실행용)

### 두 가지 모드

#### 모드 1: `verify` (기본)

**언제 쓰는가**: 이미 동작하는 코드에 테스트를 추가할 때.

**자연어 호출 패턴**:
- `goods-order.service.ts에 테스트 추가해줘`
- `GoodsOrderService 테스트 짜줘`
- `이 파일에 테스트 없는데 만들어줘`
- `calculateRefund 함수 테스트 작성해줘`
- `auth 모듈 테스트 커버리지 높여줘`

**매칭 키워드**: `테스트 추가`, `테스트 짜`, `테스트 만들`, `테스트 작성`, `커버리지 높여`

**기대 결과**: GREEN — 단, 코드를 맹신하지 않는다 (아래 철학 참고)

**핵심 철학 (선택 C 적용)**:
> verify 모드는 "코드가 현재 하는 일"을 기록하는 게 아니다.
> **"코드가 해야 하는 일(이름값, should do)"을 검증**한다.
> 코드의 이름, 시그니처, 주변 코드, 도메인 맥락에서 "해야 하는 일"을 추론하고
> 그것을 테스트한다. 현재 동작과 "해야 하는 일"이 다르면 RED가 나올 수 있고,
> 그건 자기충족적 테스트의 함정을 피하는 신호이자 잠재 버그 발견의 기회다.

**워크플로우**:
1. 대상 파일을 읽는다
2. 같은 디렉토리의 기존 테스트 2~3개를 읽어 컨벤션을 학습한다
3. **트리아지**를 출력한다
4. CRITICAL/HIGH 우선으로, **"코드가 해야 하는 일"** 기준으로 테스트를 작성한다
5. `yarn test --testPathPattern=<file>`로 실행한다
6. 모든 테스트가 GREEN인지 확인한다
7. **셀프 체크 리포트**를 출력한다

**예상 밖 RED가 나온 경우** (가장 중요):
- STOP. 자동 수정하지 않는다.
- 다음 두 가능성을 사용자에게 제시:
  - (a) 테스트가 잘못 작성됐다 (에이전트가 코드 의도를 잘못 추론함)
  - (b) 코드에 잠재 버그가 있다 (에이전트가 "해야 하는 일"과 "실제 동작"의 괴리를 발견)
- 사용자 판단을 기다린다.

> 이것이 에이전트를 **단순 코드 생성기가 아니라 잠재 버그 발견 도구**로 격상시키는 핵심이다. "GREEN 나올 때까지 테스트 수정"은 금지.

#### 모드 2: `fail-first`

**언제 쓰는가**: 아직 존재하지 않거나 버그가 있는 기능에 대한 테스트를 먼저 작성할 때. TDD 또는 회귀 테스트 흐름.

**자연어 호출 패턴**:
- `주문 금액이 음수일 때 결제가 통과되는 버그, 회귀 테스트 짜줘`
- `웹훅 중복 처리 버그, 재현 테스트 먼저 만들어줘`
- `결제 3분 만료 기능, TDD로 테스트 먼저 작성해줘`
- `매칭업 랭킹 로직 스펙 테스트 먼저 짜줘`
- `이 함수 실패부터 만들어줘`

**매칭 키워드**: `회귀`, `regression`, `재현`, `TDD`, `스펙 먼저`, `실패부터`, `먼저 짜`, `먼저 만들`, `버그`

**기대 결과**: RED (아직 코드가 올바르지 않으므로)

**핵심 철학**:
> fail-first의 본질은 **"코드가 아프다고 전제하고 그 증거를 만드는 것"**이다.
> 회귀 테스트, 버그 재현, TDD 모두 같은 원리다 — "지금 시점에서 실패해야 하는
> 테스트를 작성하고, 실패를 확인한 다음, 사용자에게 치료를 맡긴다."
> 새 기능을 만들 때 이 모드로 시작하면 **테스트가 명세가 되고, 코드가 명세를 구현**하게 된다.

**워크플로우**:
1. 관련 코드를 읽는다 (있다면)
2. **트리아지**를 출력한다
3. 원하는 행동을 정의하는 테스트를 작성한다 (현재 코드에서 실패해야 함)
4. `yarn test`로 실행한다
5. 테스트가 RED인지 확인한다
6. **핸드오프 리포트**를 출력한다
7. **STOP**. 프로덕션 코드를 작성하지 않는다. GREEN까지 진행하지 않는다.

**예상 밖 GREEN이 나온 경우**:
- STOP. 기대한 실패가 나오지 않았음을 보고.
- 두 가능성 제시:
  - (a) 테스트가 실제로 버그/누락 기능을 건드리지 않음
  - (b) 버그가 이미 수정됐거나 기능이 이미 존재함
- 사용자 판단을 기다린다.

### 핸드오프 원칙 (fail-first 모드의 핵심)

fail-first 모드에서 RED를 확인한 후, **프로덕션 코드 수정은 사용자의 몫**이다. 에이전트는:
- 어느 파일을 수정해야 하는지 **힌트**를 준다 (한 줄 수준)
- 실제 코드는 작성하지 않는다
- 사용자가 수정 후 직접 `yarn test`로 GREEN을 확인하거나, 메인 Claude Code 세션에 구현을 위임한다

이유: 에이전트 이름이 `test-writer`다. 구현 작성까지 맡기면 책임 경계가 흐려지고 통제력이 약해진다.

### 강제 행동 (반드시 한다)
- happy path마다 **실패 케이스 1개 + 경계 케이스 1개** 추가
- 상태 변경 작업은 **멱등성** 검증
- 권한 체크 코드는 **권한 없는 케이스** 검증
- 외부 연동은 **실패/타임아웃 모드** 검증
- 트리아지는 항상 먼저 출력
- 기존 테스트 2~3개를 먼저 읽고 컨벤션 따름

### 금지 행동 (절대 안 한다)
- private 메서드 직접 테스트
- 통제 가능한 내부 객체를 mock
- `skip`/`xit`를 이유 설명 없이 남기기
- 한 문장으로 설명 못 하는 테스트 작성
- 중요한 필드에 `expect.any(...)` 사용
- 테스트 안에 `if`/`for`로 여러 시나리오 분기 (단, `test.each`는 허용)
- **프로덕션 코드 수정** (테스트 파일만)
- **예상 밖 결과 자동 수정** (RED/GREEN 불일치 시 멈추고 보고)

### 출력 형식

**트리아지 (항상)**:
```
## Triage: <ModuleName>

[CRITICAL] <메서드> — <실패 시 영향>
[HIGH]     <메서드> — <실패 시 영향>
[MEDIUM]   <메서드> — <실패 시 영향>
[LOW]      <메서드> — <실패 시 영향>

Test focus: CRITICAL과 HIGH. MEDIUM은 여유 있을 때. LOW는 명시적 요청 시.
```

**셀프 체크 리포트 (verify 모드)**:
```
## Self-Check

Tests added: N
Tests passing: N/N

신념 준수:
- [OK] 트리아지 수행, CRITICAL/HIGH 경로 커버
- [OK] happy path마다 실패 케이스 포함
- [OK] 경계 케이스 포함
- [OK / N/A] 멱등성 검증
- [OK / N/A] 권한 검증
- [OK / N/A] 외부 실패 모드 검증
- [OK] 한글 테스트 이름 (또는 프로젝트 컨벤션)
- [OK] 내부 객체 mock 없음
- [OK] 결정적 (실제 시간/랜덤/네트워크 없음)

커버하지 않은 케이스 (이유 포함):
- <케이스>: <이유>

후속 권장:
- <다음에 추가하면 좋은 것>
```

**핸드오프 리포트 (fail-first 모드)**:
```
## Handoff (fail-first 완료)

Test file: <경로>
작성한 테스트:
- it('<이름>')

Status: RED (기대대로)

실패 상세:
  At:       <파일/함수>
  Expected: <테스트가 기대한 것>
  Actual:   <실제 결과>

다음 액션 (사용자):
  1. <파일>을 <한 줄 방향>으로 수정
  2. yarn test --testPathPattern=<패턴>
  3. GREEN 확인

참고: 에이전트는 여기서 멈춤. 코드 수정은 사용자 몫.
```

## 5. `test-reviewer` 에이전트

### 목적
기존 테스트 파일을 감사하고 **부족한 케이스와 안티패턴을 리포트**한다. **코드는 절대 수정하지 않는다.**

### 도구 권한
`Read`, `Glob`, `Grep` **만**.

`Edit`, `Write`, `Bash` 없음 → 부수효과 0, 언제든 안전하게 호출 가능.

### 언제 쓰는가
- PR 올리기 직전: "내 테스트 충분한지 봐줘"
- 코드 리뷰 단계: "이 PR 테스트 검토해줘"
- 주기적 감사: "이 모듈 테스트 커버리지 점검해줘"
- 리팩터링 후: "테스트가 여전히 행동을 잘 검증하는지 봐줘"

### 워크플로우
1. 대상 테스트 파일을 읽는다
2. 대응하는 소스 파일을 읽는다 (무엇을 검증해야 하는지 파악)
3. 같은 디렉토리의 다른 테스트 2~3개를 읽는다 (컨벤션 학습)
4. 소스에 대한 **트리아지**를 수행한다
5. 기존 테스트가 트리아지의 어느 항목을 커버하고 어느 항목을 누락했는지 교차 검증한다
6. 신념 카드 7개를 체크리스트로 적용한다
7. **추가 감지**를 수행한다 (아래)
8. 카테고리별 리포트를 생성한다

### 추가 감지 항목 (신념 카드 외)

사람이 놓치기 쉽고 정적 분석으로 에이전트가 잘 잡을 수 있는 항목들:

#### Flaky 위험
- `Date.now()`, `new Date()` 직접 사용
- `Math.random()`
- 실제 `setTimeout`, `setInterval`
- 실제 네트워크 호출
- 외부 시간/상태에 의존하는 단언

#### 순서 의존성
- `beforeAll`에서 만든 데이터를 여러 테스트가 공유
- 공유 mutable state
- 테스트 실행 순서를 바꾸면 깨지는 구조

#### 의도가 같은 중복 테스트
- 이름은 다르지만 같은 행동을 검증
- 한 경로를 두 테스트가 모두 커버
- 통합 테스트와 단위 테스트가 같은 케이스를 중복 검증

### 엄격 규칙
- **수정 도구 없음**: 어떤 파일도 건드리지 않는다
- **테스트 실행 안 함**: Bash 권한 없음
- **구체적이지 않으면 포함하지 않음**: 파일/라인/구체적 행동으로 고정할 수 없으면 리포트에 넣지 않는다
- **피상적 칭찬 금지**: "잘 짜셨네요" 같은 내용 없는 문장 금지
- **스타일 비판 금지**: 프로젝트가 이미 쓰는 스타일은 비판하지 않는다

### 출력 형식

**빈 섹션은 생략한다.** "발견된 안티패턴 없음" 같은 placeholder 금지.

```
## Triage: <ModuleName>

이 모듈의 핵심 위험 경로 (1~3줄):
- <경로 1>: <실패 시 영향>
- <경로 2>: <실패 시 영향>

## Missing Cases

[CRITICAL] <케이스 설명> — 위반 신념: #N
   제안 테스트: it('<구체적 테스트 이름>') — <구체적 검증 내용>

[MEDIUM]   <케이스 설명>
   제안 테스트: ...

[LOW]      <케이스 설명>
   제안 테스트: ...

## Anti-Patterns

- <구체적 문제> at <파일:라인>
  왜 문제인가: <한 줄>
  제안 수정: <한 줄>

## Flaky Risks

- <구체적 위험> at <파일:라인>

## Duplicate / Overlapping Tests

- it('...') at <파일:라인>과 it('...') at <파일:라인>이 같은 행동 검증

## Recommended Next Actions

1. <최우선 — 보통 CRITICAL 케이스>
2. <다음 우선순위>
3. ...

writer에 넘길 명령어 제안:
  "<사용자가 복붙할 수 있는 문장>"
```

### 구체성 요구사항

모든 지적은 **구체적이고 실행 가능**해야 한다. 금지:
- ❌ "엣지 케이스 부족" → ✅ "`createOrder()`의 `amount = 0` 케이스 누락"
- ❌ "mock이 많음" → ✅ "42라인에서 `OrderRepository`를 mock 중인데, 이는 내부 협력자로 `mongodb-memory-server`로 진짜 구성해야 함"
- ❌ "이름이 불명확" → ✅ "23라인 `it('test 1')` → 행동을 서술하게 변경, 예: `it('주문 금액이 음수면 거부한다')`"

## 6. 두 에이전트의 상호작용

```
[사용자 시작]
       ↓
[1] test-reviewer 호출 (안전, 부수효과 0)
       ↓
[2] 리포트 출력 — 부족한 케이스, 안티패턴, 권장 액션
       ↓
[3] 사용자가 리포트 읽고 우선순위 결정
       ↓
[4] test-writer 호출 (의도적, 리포트에서 제안한 명령어 복붙)
       ↓
[5] writer가 verify 또는 fail-first 모드로 동작
       ↓
[6] 코드 작성 + 실행 + 리포트
       ↓
[7] 사용자가 결과 확인, 필요 시 다시 reviewer로 검증
```

**핵심 원칙**:
- reviewer는 안전하니 자주 호출 OK
- writer는 명확한 의도가 있을 때만
- 사용자가 중간에서 판단 — 에이전트가 자율적으로 체이닝하지 않음

## 7. 사용 예시 (자연어 호출)

> 사용자는 에이전트 이름을 몰라도 된다. Claude Code가 요청 문구를 보고 description 매칭으로 알아서 라우팅한다.

### 예시 1: 기존 코드에 테스트 추가 (verify)
```
goods-order.service.ts에 테스트 추가해줘
```
→ verify 모드 → 트리아지 → 작성 → 실행 → GREEN 확인 → 셀프 체크

### 예시 2: 기존 테스트 감사 (reviewer)
```
goods-order.service.spec.ts 테스트 리뷰해줘
```
→ 리포트만 출력 (수정 없음)

### 예시 3: 버그 회귀 테스트 (fail-first)
```
주문 금액이 음수일 때 결제가 통과되는 버그가 있어. 회귀 테스트 짜줘
```
→ fail-first 모드 → RED 테스트 작성 → 핸드오프 리포트 → 사용자가 버그 수정

### 예시 4: 새 기능 TDD (fail-first)
```
결제 만료 시간을 3분으로 제한하는 기능 추가할 건데, TDD로 테스트 먼저 짜줘
```
→ fail-first 모드 → 스펙 기반 테스트 → RED 확인 → 핸드오프 → 사용자가 코드 작성

### 예시 5: reviewer → writer 체이닝
```
1단계: auth.service.spec.ts 테스트 충분한지 봐줘
       → "CRITICAL: 만료된 토큰 케이스 누락..." 리포트

2단계: auth.service.spec.ts에 만료된 토큰 케이스 추가해줘
       → verify 모드로 추가 후 GREEN 확인
```

### 예시 6: 커밋 전 자동 리뷰 (/safe-commit)
```
/safe-commit
```
→ 변경된 테스트 파일에 대해 test-reviewer 자동 호출 → 리포트 → CRITICAL 없으면 커밋 진행

### 예시 7: 새 기능 TDD 전체 흐름
```
[새 기능 시작]
1. "결제 만료 기능, TDD로 테스트 먼저 짜줘"
   → fail-first 모드 → RED 확인 → 핸드오프
2. 사용자가 코드 작성 (직접 또는 메인 Claude에 위임)
3. yarn test → GREEN 확인
4. /safe-commit → reviewer가 테스트 검증 → 커밋
```
> 이것이 **"테스트 먼저, 코드 나중"** 의 습관을 강제하는 권장 워크플로우다.

### 모드 자동 감지 우선순위

에이전트가 요청에서 모드를 결정하는 순서:

1. 요청에 `회귀`, `regression`, `재현`, `버그`, `TDD`, `스펙 먼저`, `실패부터`, `먼저 짜`, `먼저 만들` 중 하나가 있는가? → **fail-first**
2. 요청에 `테스트 추가`, `테스트 짜`, `테스트 만들`, `커버리지 높여` 등이 있는가? → **verify**
3. 애매하면 → 사용자에게 한 번만 확인 후 진행

## 8. 모델 선택: 왜 Opus?

두 에이전트 모두 **Opus 4.6**.

### 이유
- **writer의 가치**: happy path를 짜는 것이 아니라 "무엇이 깨질 수 있는가를 상상하는 것". 이건 추론 과업이고, Opus가 명백히 우수.
- **reviewer의 가치**: "보이지 않는 것을 보는 것". 부족한 케이스, 안티패턴, 미묘한 결합 감지. 100% 추론 깊이의 함수.
- **도메인 복잡도**: 결제/주문/큐/동시성 같은 영역에선 Sonnet과 Opus의 결과 품질 차이가 체감된다.
- **비용**: 한 번 호출에 쓰는 토큰이 수천 단위라 절대 비용이 부담스럽지 않음.

### 언제 Sonnet으로 내릴 만한가
- 단순 CRUD/유틸 테스트만 짜는 경우
- 비용에 매우 민감한 환경
- 연속 작성 등 빠른 피드백이 중요한 경우

전환 방법: 각 에이전트 파일의 frontmatter에서 `model:` 한 줄만 바꾸면 됨.

## 10. 디자인 원칙 요약

| 원칙 | 적용 |
|---|---|
| 책임 분리 | writer는 작성, reviewer는 리포트. 겹치지 않음 |
| 부수효과 격리 | reviewer는 read-only, writer는 명시적 호출 시만 |
| 신념의 코드화 | 사람의 의지력 대신 시스템 프롬프트로 강제 |
| 자동 수정 금지 | 예상 밖 결과는 사용자 판단 맡김 (잠재 버그 발견 도구) |
| 컨벤션 우선 | 프로젝트가 쓰는 스타일이 에이전트 선호를 이김 |
| 한 번에 끝내기 | 에이전트는 사용자 확인 없이 진행. 트리아지가 체크포인트 |
| 구체성 강제 | 모든 지적과 제안은 파일/라인/구체 행동에 고정 |

## 11. 자동화: 전역 규칙 + `/safe-commit`

### A. `~/.claude/CLAUDE.md` 전역 규칙

다음 규칙이 전역 CLAUDE.md에 추가된다:

1. **커밋 전 리뷰**: 테스트 파일(`*.spec.ts`, `*.test.ts`)이 변경된 커밋을 만들기 전에 `test-reviewer`를 호출해 리포트를 확인한다. CRITICAL 항목이 있으면 사용자 확인 없이 커밋하지 않는다.
2. **새 기능 시 TDD 권장**: 새 기능/엔드포인트/서비스를 작성할 때는 `fail-first` 모드로 테스트를 먼저 작성하는 것을 권장한다. 사용자가 "코드 먼저 짜고 싶다"고 명시하지 않는 한 TDD 흐름을 제안한다.

### B. `/safe-commit` 슬래시 명령

커밋을 안전하게 수행하는 슬래시 명령이다. 워크플로우:

1. `git diff --cached`로 staged 변경 확인
2. 변경된 테스트 파일이 있으면 → `test-reviewer` 자동 호출
3. 리포트 출력
4. CRITICAL 항목이 있으면 → "CRITICAL 항목이 N개 있습니다. 그래도 커밋할까요?" 확인
5. CRITICAL 없거나 사용자 승인 → `git commit` 진행
6. 변경된 테스트 파일이 없으면 → reviewer 없이 바로 커밋 진행
