# 전역 규칙

## 테스트 품질 가드레일

1. **커밋 전 리뷰**: 테스트 파일(`*.spec.ts`, `*.test.ts`)이 변경된 커밋을 만들기 전에 `test-reviewer`를 호출해 리포트를 확인한다. CRITICAL 항목이 있으면 사용자 확인 없이 커밋하지 않는다.

2. **새 기능 시 TDD 권장**: 새 기능/엔드포인트/서비스를 작성할 때는 `fail-first` 모드로 테스트를 먼저 작성하는 것을 권장한다. 사용자가 "코드 먼저 짜고 싶다"고 명시하지 않는 한 TDD 흐름을 제안한다.

3. **테스트 작성 — test-writer 사용 기준**: 다음 중 **하나라도 해당**하면 `test-writer` 서브에이전트로 작성. 모두 미해당이면 메인 컨텍스트에서 직접 Edit 으로 작성.
   - 신규 spec 파일 생성 (0 → 1) — 패턴/구조 설계 필요
   - TDD fail-first 모드 — 사용자가 "테스트 먼저", "회귀 재현", "버그 reproduce" 명시
   - 신규 케이스 5개 이상 — 일정 규모 이상은 격리 작성이 깨끗
   - 새 mock 구조 / cross-module 셋업 — 기존 spec 패턴으로 안 풀리는 케이스

   기준 미해당 예: 기존 spec 에 1~4 케이스 추가, 기존 mock 재활용, 단순 회귀 보강. 임계치(케이스 5개)는 시작점 — 사용 패턴 보면서 조정한다.

   ※ 위 기준과 무관하게 **커밋 전 test-reviewer 호출은 항상 의무** (위 1번).

4. **필수 플로우는 seam-seed 없이 end-to-end 검증**: "반드시 동작해야 하는 흐름"(생산자→소비자 체인, 예: 큐 적재→워커 처리)은 통합 테스트에서 *체인이 생산해야 할 중간 상태를 손으로 seed하지 말고* 시작점만 만든 뒤 실제 서비스 호출로 끝까지 돌려 최종 관찰 상태를 단언한다. 중간 상태(예: `status:'processing'`·할당된 처리자)를 직접 박으면 그 이음새 버그(생산자가 그 상태를 못 만드는 경우)가 구조적으로 안 잡힌다. 피처마다 이런 플로우를 최소 1개 식별해 둔다. (단위 테스트에서 소비자 로직만 검증할 때의 seed는 허용 — 이 규칙은 "그 체인의 end-to-end 테스트가 하나도 없음"을 막는 것.)

## 배포 안전 가드레일

1. **배포 영향 감지**: migration, 환경변수, config 변경 등 배포에 영향을 주는 변경을 작업 중 감지하면, PR 생성 시 본문 체크리스트에 자동으로 포함한다. 사용자가 별도로 요청하지 않아도 수행한다.

2. **PR 기반 배포**: main 브랜치는 PR 머지로만 배포한다. 로컬에서 main에 직접 머지 + push하지 않는다.

## 선택 4단계 (필수 포맷)

기술 선택·설계 의사결정·작업 흐름·구현 방식 등 무언가를 "고른다"는 응답은 다음 4단계로 답한다. 단 **선택의 무게(비가역성·blast radius·비용)에 비례해 깊이를 조절**한다 — 가벼운 선택(변수명, 이미 쓰는 lib 중 택1, 자명한 것)은 한 줄 결론으로 끝내고, 무게 있는 선택(아키텍처·스키마·라이브러리 도입·마이그레이션)만 4단계 풀로 펼친다. 무게 있는 선택에서 단일 선택지만 결론부터 제시하는 응답 금지.

1. **후보군**: 고려한 대안들을 최소 2개, 가능하면 3~5개 펼친다. lifecycle / layer / 책임 단위로 범주를 다양화한다 (예: 표준 위치 / 안전망 / 사후 처리 / 미들웨어 / 글로벌 등).
2. **각 후보의 트레이드오프**: 장점·단점·위치·책임·비용을 한 줄씩이라도 비교한다. 표 형태가 좋다.
3. **택한 것**: 명시적으로 하나 지목한다.
4. **택한 이유 + 한계**: 왜 이 맥락에서 다른 것이 아니라 그것인가. 이 선택으로 못 막는 케이스는 무엇이고, 그건 어떤 안전망 / 다음 단계로 보강하는가까지.

**금지**:
- 후보군 펼치지 않고 결론부터 제시
- "이게 표준이라서" / "관례상" / "보통 이렇게 함" 같은 관성 이유
- 트레이드오프 없이 한 선택지만 제시

## right-size 선택 + 스케일 경로

선택은 **그 프로젝트의 현재 규모에서 가장 단순한 안**을 고른다. 지금 필요 없는 분산/캐시 계층/샤딩/CQRS를 미리 깔지 않는다.

- **규모 앵커는 추측 트래픽(DAU N)이 아니라 실제 용량 제약**으로 잡는다 — 인스턴스 CPU/메모리, 단일성 여부, 데이터량. "가장 먼저 깨질 축"을 식별한다.
- 무게 있는 선택(아키텍처·스키마·라이브러리 도입·마이그레이션)에만 **스케일 트리거**를 첨부: "관측 지점 X(CPU% / P95 / 메모리% / 문서 수)를 넘으면 _이렇게_ 깨진다 → 그때 _이걸로_ 진화". 이게 위 4단계의 "한계 + 안전망/다음 단계"를 채우는 방식이다.
- **진화 경로는 tier 이름에서 멈추지 않는다**: 무게 있는 선택이면 스케일 트리거의 "그때 _이걸로_ 진화" 를 제시할 때 항상, 그 전환이 (a) **시스템 토폴로지를 어떻게 바꾸는지** + (b) **코드의 어느 지점이 어떻게 바뀌는지**(멱등성 키·락 래핑·projection 분리·outbox 발행 등)를 각각 한 줄 이상 명시한다. tier 이름(예: "이벤트 기반")만 적고 끝내는 응답 금지. 즉 한 사이클 = 선택 4단계 → right-size 앵커 → 스케일 트리거 → 진화 경로(tier) → **진화 delta(토폴로지 변화 + 코드 변화)** 까지 닫는다.
- 가벼운 선택은 한 줄 결론 — 스케일 분석을 기계적으로 붙이지 않는다(설명도 right-size).
- **무게 강제 키워드**: 사용자가 "무겁게" (변형: "무겁게 봐줘" / "무게 있게" / "무게 있게 봐줘") 라고 하면, 결정의 성격이 가벼워 보여도 그 선택을 무게 있는 선택으로 취급해 4단계 풀(후보 비교 포함)로 펼친다. ("상세하게/자세히" 는 단순 설명 요청이므로 무게 강제 키워드가 아니다.) 진화 delta(토폴로지+코드 변화)만 따로 강제하려면 "스케일 경로까지" / "코드 delta까지" 를 쓴다.
- 스케일 사다리 프레임: **T0** 단일/소규모 = 단순함 최우선 → **T1** 수평확장 = 분산락·멱등성·인덱스·캐시 → **T2** 읽기부하 = replica 분리·캐시 계층화·팬아웃 전환 → **T3** 쓰기한계 = 이벤트 기반·도메인 분리/샤딩.
- 임계치 숫자는 placeholder — **각 프로젝트의 실측/용량 역산으로 교정**해서 쓴다.

## robustness 메커니즘 추가 가드

retry·분산락·캐시·큐 같은 robustness/인프라 메커니즘을 추천하기 전에 반드시:

1. **강제조건 확인**: 그 메커니즘이 필요해지는 조건(경합 빈도·규모·실패모드)이 *지금 이 맥락에서* 실제로 성립하는가? 기존 코드에 같은 패턴이 있다는 이유(관성)로 전이 금지 — 그 패턴이 *왜* 존재하는지(강제조건)를 먼저 확인하고, 동일 조건이 성립할 때만 적용한다. 추천 근거에 "코드베이스에 있으니"가 들어가면 red flag → 강제조건으로 재정당화하거나 폐기.
2. **safety vs liveness 분리**: 필수(safety: 정합성·이중과금 방지)와 선택(liveness/UX: 경합 시 성공 보장)을 나눠 **필수만** 메커니즘을 추가한다. 선택은 발생확률·비용이 둘 다 낮으면 보류하고 scale-trigger로 남긴다. (예: 저경합 유료 액션의 동시요청은 안전성만 보장하고, transient 실패는 사용자 재시도로 흡수.)

## API 설계 가드

- **개인화 값(`is_liked`·좋아요·`followed_by_me` 등)은 절대 캐시된 payload 안에 넣지 않는다.** 캐시는 익명·공유 가능한 것만 담고, 개인화는 캐시 밖에서 매 요청 fresh로 붙인다. (전역 캐시면 타 유저 노출, 유저별 키면 엔트리 폭발)
- 새 조회 API/엔드포인트를 설계할 때(합칠지/병렬로 나눌지/캐시 위치/API 분리 판단) `api-composition-caching` 스킬을 참조한다.

## 멀티-태스크 실행 — 태스크 완료 요약 (2레이어)

subagent-driven-development 등 멀티-태스크 플랜을 연속 실행할 때, **각 태스크가 끝나는 시점**(커밋 + 스펙/품질 리뷰 통과 직후, 다음 태스크 dispatch 전)에 그 태스크 요약을 **두 레이어로 함께** 제공한다. (공유 플러그인 스킬은 안 건드리고 이 전역 규칙으로 오버라이드한다.)

1. **코드 레벨 (변경 + 이유)**: 커밋 SHA · 변경/신규 파일 · 테스트·빌드 결과 · 다음 태스크로 넘기는 flag(인계 사항). **각 핵심 코드 변경마다 "왜 이렇게 바꾸거나 추가했는지"(설계 이유 / 대안 대비 / 왜 여기인지)를 한 줄씩 덧붙인다** — 무엇을 바꿨는지뿐 아니라 *왜 그 방식인지*.
2. **한국어 설명**: 각 변경이 "무슨 작업이었는지"를 도메인 평어로 — 코드 용어가 아니라 의도/효과 중심. (예: "승인 끝난 게시물을 작성자가 포인트를 내고 다시 검수 대기열에 올리는 진입점을 만들었다" 식)

코드 레벨만도, 한국어 설명만도 아닌 **둘 다**. 연속 실행이라도 태스크 단위로 끊어 사용자가 리뷰할 수 있게 한다.
